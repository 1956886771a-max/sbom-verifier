# 项目范围定义（当前版本）

## 1. 项目名称

SBOM 真实性验证系统（第一阶段：npm 单生态深耕）

---

## 2. 当前阶段的核心目标

本项目当前的首要目标是：

> 构建一个面向 npm 生态的 SBOM 真实性验证研究原型，并支撑第一篇工程型 EI 论文投稿。

当前阶段以“**论文可成立**”为第一优先级，而不是追求完整产品化功能。

---

## 3. 当前研究问题

本阶段拟解决的核心问题是：

> 如何验证一个 npm 项目的 SBOM 是否真实反映其声明依赖与实际安装/打包组成，并识别其中的不一致类型？

具体包括：

1. 如何从声明文件中提取项目依赖信息
2. 如何从 SBOM 中解析组件信息
3. 如何从实际安装目录或构建产物中提取真实存在的组件
4. 如何对多来源依赖信息进行统一表示与匹配
5. 如何检测并分类不一致
6. 如何尽量降低误报，使结果适合实验与论文写作

---

## 4. 当前阶段必须完成的内容

以下内容属于第一篇论文主体的“必须完成项”：

### 4.1 npm 生态支持
- 支持解析 `package.json`
- 支持解析 `package-lock.json`
- 支持提取声明依赖集合
- 能够处理 npm 生态中的基础版本信息

### 4.2 SBOM 处理
- 支持生成或读取 SBOM
- 支持至少一种主流 SBOM 格式的解析
- 能将 SBOM 中的组件信息转成统一内部表示

### 4.3 实际依赖提取
- 支持从 `node_modules` 中提取实际安装依赖
- 如有必要，可辅助扫描构建产物中的组件信息
- 建立“实际依赖集合”作为对照来源

### 4.4 差异检测
- 支持检测以下基础不一致：
  - 声明存在但实际未发现
  - 实际发现但声明缺失
  - 同名组件版本不一致
- 输出结构化差异结果

### 4.5 基础误报控制
- 初步考虑以下因素带来的误报：
  - `devDependencies`
  - `optionalDependencies`
  - 传递依赖
  - npm 安装扁平化
  - 版本范围表达差异

### 4.6 实验与论文支撑
- 准备真实 npm 项目样本
- 跑通批量实验
- 统计基础指标
- 提供若干典型案例分析
- 支撑第一篇 EI 论文写作

---

## 5. 当前阶段的加分项

以下内容如果时间允许可以做，但不是当前阶段的硬性要求：

- PyPI 生态的初步支持
- Maven 生态的初步支持
- 更复杂的依赖层次重建
- 一致性评分模型
- 更完善的异常分类体系
- 更漂亮的命令行输出
- 实验自动化脚本增强

---

## 6. 当前阶段明确暂不优先的内容

以下内容当前阶段原则上不作为主线任务：

### 6.1 ML 增强
暂不优先进行：
- 机器学习分类
- 风险排序模型
- 异常智能预测
- LLM 自动解释

原因：第一篇论文当前以基础验证系统与实验成立为主，ML 可作为后续扩展方向。

### 6.2 前端系统
暂不优先进行：
- Web 前端页面
- 可视化平台
- 用户管理
- 在线上传与检测界面

原因：当前以研究原型与后端闭环为主，前端适合后续毕业设计阶段增强。

### 6.3 动态运行时监测
暂不优先进行：
- 复杂动态加载追踪
- 运行时依赖监测
- 动态插桩分析

原因：复杂度较高，不利于当前阶段快速形成论文主体。

### 6.4 过早多生态深耕
暂不在第一阶段一开始就同步深做多个生态。  
原则上先保证 npm 做深、做稳，再视时间扩展到第二生态。

---

## 7. 当前阶段的技术路线

本阶段采用以下基本技术路线：

1. 从 npm 项目声明文件中提取依赖信息
2. 生成或读取对应 SBOM
3. 从 `node_modules` 或产物中提取实际依赖
4. 构建统一的内部依赖表示
5. 进行名称、版本和组件匹配
6. 检测并分类不一致
7. 输出实验结果与分析报告

---

## 8. 当前阶段的交付物

本阶段预期交付包括：

### 系统层面
- 一个可运行的研究原型
- 基础 CLI 或脚本式运行流程
- 若干实验脚本与中间结果

### 论文层面
- 研究问题定义
- 方法描述
- 系统设计图
- 实验设计与结果
- 第一篇 EI 论文初稿

### 数据层面
- 一批 npm 项目样本
- 对应 SBOM、中间扫描结果和差异报告
- 若干可用于论文 case study 的示例

---

## 9. 当前阶段的成功标准

如果满足以下条件，则认为第一阶段目标基本达成：

1. npm 单生态验证闭环可以稳定运行
2. 至少支持基础差异检测与初步误报控制
3. 能在一批真实项目上完成实验
4. 能产出论文所需的图表、案例与指标
5. 能形成一篇结构完整、可投稿的 EI 工程型论文初稿

---

## 10. 当前阶段的执行原则

### 原则 1：优先论文主体成立
任何新功能都应首先判断是否直接增强第一篇论文主体。

### 原则 2：优先最小可运行闭环
先跑通，再增强；先完成，再优化。

### 原则 3：优先 npm 深耕
第一阶段重点是“做深”，不是“做广”。

### 原则 4：边做边沉淀论文内容
实现、实验、文档与论文写作同步推进。

### 原则 5：避免功能失控
除非明确有助于第一篇论文，否则暂缓额外扩展。

---

## 11. 后续可扩展方向（第二阶段以后）

待第一阶段主体完成后，可进一步扩展：

- 第二生态支持（如 PyPI）
- 多生态统一验证框架
- ML 增强异常分类与风险排序
- Web 前端与可视化平台
- CI/CD 集成
- 更大规模实验与数据集建设

---

## 12. 当前结论

当前阶段的最优策略是：

> 以 npm 单生态深耕为主线，优先完成一个能够支撑 EI 论文投稿的 SBOM 真实性验证系统原型。
